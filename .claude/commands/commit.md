# 변경사항 커밋

이 세션에서 수행된 변경사항에 대한 git 커밋을 생성하는 것이 당신의 임무입니다.

## 프로세스:

1. **무엇이 변경되었는지 생각하세요:**
   - 대화 기록을 검토하고 무엇이 달성되었는지 이해하세요
   - `git status`를 실행하여 현재 변경사항을 확인하세요
   - `git diff`를 실행하여 수정사항을 이해하세요
   - 변경사항이 하나의 커밋이어야 하는지 아니면 여러 논리적 커밋이어야 하는지 고려하세요

2. **커밋을 계획하세요:**
   - 어떤 파일들이 함께 속하는지 식별하세요 (프론트엔드/백엔드 분리 고려)
   - 명확하고 설명적인 커밋 메시지를 초안으로 작성하세요
   - 커밋 메시지에서 명령형을 사용하세요
   - 무엇이 변경되었는지뿐만 아니라 왜 변경되었는지에 집중하세요
   - 기능 영역별로 그룹화를 고려하세요 (scheduler, notion, UI 등)

3. **사용자에게 계획을 제시하세요:**
   - 각 커밋에 추가할 파일들을 나열하세요
   - 사용할 커밋 메시지를 보여주세요
   - 변경사항이 프론트엔드, 백엔드, 또는 둘 다에 영향을 미치는지 표시하세요
   - 물어보세요: "이러한 변경사항으로 [N]개의 커밋을 생성할 계획입니다. 진행해도 될까요?"

4. **확인 후 실행하세요:**
   - 특정 파일과 함께 `git add`를 사용하세요 (`-A`나 `.`는 절대 사용하지 마세요)
   - 계획된 메시지로 커밋을 생성하세요
   - `git log --oneline -n [number]`로 결과를 보여주세요

## 중요사항:

- **공동 작성자 정보나 Claude 속성을 절대 추가하지 마세요**
- 커밋은 사용자만이 작성해야 합니다
- "Claude로 생성됨" 메시지를 포함하지 마세요
- "Co-Authored-By" 라인을 추가하지 마세요
- 사용자가 작성한 것처럼 커밋 메시지를 작성하세요

## 기억하세요:

- 이 세션에서 수행된 것의 전체 컨텍스트를 가지고 있습니다
- 관련 변경사항을 함께 그룹화하세요 (프론트엔드/백엔드, 기능 영역)
- 가능한 한 커밋을 집중적이고 원자적으로 유지하세요
- 모노레포 구조를 고려하세요 (packages/frontend, packages/backend)
- 사용자가 당신의 판단을 신뢰합니다 - 커밋을 요청했기 때문입니다

## 커밋 메시지 패턴 예시:

- `feat: 새로운 가사 카테고리 기능 추가`
- `fix: TypeORM 엔티티 관계 문제 해결`
- `refactor: 스케줄러 서비스 구조 개선`
- `docs: README의 API 문서 업데이트`
- `style: prettier로 코드 포맷팅`
- `test: 스케줄러 모듈에 대한 단위 테스트 추가`
